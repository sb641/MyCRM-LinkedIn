# MyCRM-LinkedIn: Обзор бизнес-логики и архитектуры

Этот документ предназначен для бизнес-аналитиков и технических менеджеров с целью дать полное представление о проекте, его целях, текущей реализации и потенциальных направлениях развития.

---

## 1. Концепция и бизнес-цели

**Основная цель:** Создать локальное CRM-приложение (local-first) для управления перепиской в LinkedIn. Приложение должно помогать пользователю отслеживать статусы общения с контактами, вовремя отправлять фоллоу-апы и автоматизировать рутинные задачи, сохраняя при этом полный контроль над данными, так как они хранятся на компьютере пользователя.

**Ключевая ценность:**

*   **Приватность и владение данными:** Все данные (контакты, сообщения, настройки) хранятся локально.
*   **Автоматизация рутины:** Помощь в написании черновиков сообщений и напоминания о необходимости связаться с контактом.
*   **Контролируемый процесс:** Пользователь полностью контролирует процесс отправки сообщений и синхронизации данных.

## 2. Ключевые бизнес-сущности и логика

Система оперирует несколькими основными концепциями:

### 2.1. Контакты и Статусы Отношений

*   **Контакт:** Центральная сущность, представляющая человека в LinkedIn.
*   **Статус Отношений (elationship_status):** Описывает текущий этап взаимодействия. Жизненный цикл контакта управляется этими статусами:
    *   
ew: Новый контакт, взаимодействие не начато.
    *   waiting_reply: Отправлено сообщение, ожидается ответ.
    *   eplied: Контакт ответил.
    *   ollowup_due: Настало время для отправки следующего сообщения (фоллоу-апа).
    *   stale: Давно не было взаимодействия.

### 2.2. Inbox и Логика Фоллоу-апов

*   **Inbox:** Основной интерфейс, отображающий список контактов. Сортировка по умолчанию — по "нуждающимся во внимании" (
eeds-attention), то есть контактам со статусом ollowup_due или eplied.
*   **Движок фоллоу-апов:** Система автоматически переводит контакт в статус ollowup_due по истечении заданного в настройках количества дней (ollowup_days) после последней отправки сообщения.

### 2.3. Генерация и отправка сообщений

Процесс отправки сообщения состоит из нескольких шагов, чтобы обеспечить полный контроль пользователя:

1.  **Цель (Goal):** Пользователь пишет краткую цель для сообщения (например, "узнать о решении по предложению").
2.  **Генерация черновика (AI Draft):** Система (используя mock-адаптер для Gemini) генерирует несколько вариантов текста сообщения.
3.  **Утверждение (Approval):** Пользователь выбирает, редактирует и утверждает один из вариантов.
4.  **Постановка в очередь (Queue Send):** Утвержденный черновик не отправляется мгновенно, а ставится в очередь на отправку как фоновая задача (send_message). Это предотвращает случайные отправки.
5.  **Отправка:** Фоновый процесс (Worker) забирает задачу из очереди и выполняет ее. *На данный момент реальная отправка через браузер не реализована, используется фейковый провайдер.*

### 2.4. Синхронизация данных

*   **Процесс:** Синхронизация инициируется пользователем вручную. Приложение имитирует вход в LinkedIn через браузер для сбора данных.
*   **Сессия браузера (rowser_session):** Для синхронизации требуется сохраненная сессия (cookies, user-agent). Пользователь должен предварительно "залогиниться" через интерфейс, чтобы сохранить эту сессию.
*   **Фоновая задача:** Процесс синхронизации также является фоновой задачей (import_threads), которая обрабатывается воркером.

### 2.5. Настройки, Бэкапы и Восстановление

*   **Локальное хранение:** Настройки и "секреты" (API-ключи) хранятся в файлах .mycrm/settings.json и .mycrm/secrets.json в корне проекта.
*   **Бэкап:** Пользователь может экспортировать все свои данные (контакты, переписки, настройки) в один JSON-файл.
*   **Восстановление:** Можно восстановить данные из JSON-файла. Система поддерживает два режима:
    *   merge: Добавление новых и обновление существующих данных.
    *   eplace: Полная замена всех текущих данных на данные из бэкапа (требует явного подтверждения).

## 3. Техническая архитектура

Проект построен как монорепозиторий под управлением pnpm. Это позволяет переиспользовать код между разными частями приложения.

### 3.1. Структура проекта


apps/
  ├─ web/       # Frontend (Next.js) и API-маршруты
  └─ worker/    # Фоновый обработчик задач (Node.js)
packages/
  ├─ core/      # Общая логика: DTO, схемы валидации (Zod), утилиты
  ├─ db/        # Схема БД (Drizzle), репозитории для доступа к данным
  ├─ automation/ # Абстракция для работы с браузером (Playwright)
  └─ ai/        # Абстракция для работы с AI-моделями (Gemini)
docs/
  └─ ...        # Документация


### 3.2. Компоненты архитектуры

*   **Frontend (pps/web):**
    *   **Фреймворк:** Next.js 15 с App Router.
    *   **Архитектура:** Используются серверные компоненты для отрисовки страниц с данными. Взаимодействие с бэкендом происходит через API Routes (/api/...).
    *   **База данных:** Frontend **не имеет прямого доступа к базе данных**. Все запросы идут через API Routes, которые, в свою очередь, используют слой сервисов.

*   **Backend Worker (pps/worker):**
    *   **Назначение:** Обработка "тяжелых" или долгих задач в фоне, чтобы не блокировать интерфейс.
    *   **Логика:** Воркер периодически опрашивает таблицу jobs в базе данных, и если находит задачи в статусе queued, он их "захватывает" (ставит unning), выполняет и обновляет статус.

*   **База данных (packages/db):**
    *   **СУБД:** SQLite, как локальный файл (.mycrm/mycrm.sqlite). Используется встроенный в Node.js драйвер 
ode:sqlite.
    *   **ORM:** Drizzle ORM для строго типизированных запросов к БД.
    *   **Паттерн доступа:** Доступ к БД инкапсулирован внутри пакета @mycrm/db/server и разрешен только на стороне сервера (в API Routes и воркере). Это обеспечивает четкую границу между клиентом и сервером.

*   Реализована архитектура с четким разделением на frontend, worker и data-слой.
*   Добавлены регрессионные E2E-тесты (Playwright) для ключевых сценариев.

*   Добавлена базовая метрика для отслеживания производительности загрузки данных.

1.  **Валидация производительности:**

    *   **Предложение:** Использовать console.time для более гранулярных измерений в сервисах, проанализировать сложные SQL-запросы, которые генерирует Drizzle.

2.  **Инструкции по запуску в Docker:**
    *   **Задача:** Создать Dockerfile и docker-compose.yml для упрощения развертывания проекта в изолированном окружении. Это сделает запуск более предсказуемым для новых пользователей.
    *   **Предложение:** Создать два сервиса в docker-compose: web и worker, которые будут использовать общую volume для базы данных SQLite.

3.  **Подготовка чек-листа релиза:**
